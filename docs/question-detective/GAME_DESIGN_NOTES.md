# 問題偵探 — 遊戲設計探討筆記

## 設計初衷

要解決的核心問題（現代學生的三大痛點）：
1. **找重點的能力缺乏** — 學生無法從大量文字中辨識關鍵資訊
2. **長文閱讀缺乏耐心** — 注意力不足，容易放棄閱讀
3. **缺乏結構化思考引導** — 不知道如何從資訊推導出結論

### 對應的遊戲機制設計

| 痛點 | 對應機制 | 說明 |
|------|----------|------|
| 找重點 | 線索收集（點擊題幹關鍵字） | 強迫學生逐字閱讀、判斷重要性 |
| 長文耐心 | 偵探敘事包裝 + 主題沉浸 | 用遊戲動機取代學習抗拒 |
| 結構化思考 | 推理 → 舉證 → 指認答案 | 不能跳過過程直接猜答案 |

### 核心流程

```
收集線索 → 推理問答 → 舉證（指認證據） → 指認答案
```

每個環節都有存在的必要：
- **收集線索**：訓練「找重點」
- **推理問答**：引導「結構化思考」
- **舉證**：強化「證據意識」，不能憑空推論
- **指認答案**：最終驗證理解是否正確

## 待討論議題

1. 何時加入帳號系統？追蹤答題率與線索選錯統計，與教室系統
   - **決議：延後，等核心遊戲模式穩定後再加**
   - Supabase Auth 現成可用，接入成本低
   - MVP 階段不設註冊牆，避免嚇跑試玩者
   - 觸發時機：當用戶產生回訪需求（「我的進度呢？」）時上線
2. 主題 × 科目配對問題：三大主題如何覆蓋所有學科？
   - 目前構想：江湖=國文/中台歷史、Cyber=生物基因/部分地理、Classic=公民法科/英語文法
   - 困擾：多個學科找不到適合的主題歸屬
     - 理化、數學、地科天文 → 可能需要「煉金術師」風格？
     - 生物物種、生殖
     - 西洋史
     - 現代生活英語
     - 部分公民、地理
   - 核心矛盾：原想「一個遊戲一個主題」，但怕某些主題沒人玩；改成「科目分散到主題」，又面臨主題數不夠覆蓋所有科目
   - **決議：主題與科目解耦**
     - 主題 = 世界觀皮膚（視覺/台詞/氛圍），任何科目可套任何主題
     - 遊戲模式 = 玩法機制，才跟科目特性綁定
     - 學生自選喜歡的皮膚，新科目只需選適合的模式，不必發明新主題
     - 主題可慢慢擴充但不再是瓶頸
3. 遊戲模式太單一 — 需要多模式對應不同學習痛點
   - 現有：偵探模式（收集→推理→舉證→指認）
   - 新模式構想：
     a. **拆彈模式**（江湖版=解毒模式）— 痛點：閱讀太慢 → 限時倒數壓力下快速作答
     b. **誰是臥底** — 痛點：看不出錯誤在哪 → 針對「何者錯誤／何者正確」的選項辨別模式
     c. **臥底模式** — 完整設計：
        - 流程：閱讀題目 → 圈選項中的錯誤文字 → 每圈即時反饋為什麼錯 → 排除三個錯誤 → 揪出正確答案
        - 結案測驗：根據剛才圈到的錯誤，出 1-3 題蘇格拉底小問答確認學生理解
        - 資料結構：DetectiveQuestion 加 optionErrors（選項錯誤標記 + quiz）
        - 教育目標：培養「覺得哪裡怪怪的」敏感度 + 即時正回饋加深印象
     d. **解密模式**（殘卷拼讀/損毀封包）— 痛點：知道大意但細節模糊 → 關鍵字挖空，從選項池拖曳填回原位
     d. **連連看/拼圖** — 小遊戲，穿插用，配對記憶/排序邏輯
   - ~~審問室~~ — 放棄，與臥底模式本質重疊，且手機呈現差、無現成題源
   - **題源限制決議：只用會考/基測考古題**（唯一合法公開題源，不用 AI 產題）
     - 偵探、臥底、拆彈：零改造，直接用考古題
     - 解密：低改造成本 — 現有題目的關鍵線索/輔助線索位置即挖空位置，題幹結構不動，只需挑選適合的題目
     - 連連看/拼圖：量少，點綴用

## 首頁重設計

### 設計原則
- **角色驅動，非功能驅動** — 進來看到的是「人」不是選單
- **一個主行動按鈕** — 不用想，點了就玩
- **首頁本身就是遊戲的一部分**，不是遊戲外面的說明書

### 頁面結構（手機直屏優先）

```
┌──────────────────────┐
│                      │
│    [主題角色立繪]     │  佔螢幕 ~40%，切主題換角色+背景
│   「角色台詞」        │  不是標語，是角色在跟你說話
│                      │
│  ┌──────────────────┐│
│  │  ▶ 繼續調查       ││  舊用戶=接續進度 / 新用戶=「開始第一案」
│  └──────────────────┘│
│                      │
│  ┌────┐ ┌────┐ ┌────┐│
│  │偵探 │ │臥底 │ │拆彈 ││  小圖標，上線=亮，未上線=🔒
│  └────┘ └────┘ └────┘│
│                      │
├──── 底部 Tab ────────┤
│  🏠首頁  📖圖鑑  👤我的 │  收集冊、設定、早期工具收進 tab
└──────────────────────┘
```

### 要點
- 主題切換 = 換角色立繪 + 背景 + 台詞，整頁跟著變
- 模式選擇是次要的，小圖標不搶注意力
- 早期模組（台灣地圖、地方政府）降級到「👤我的」或「更多工具」tab
- 底部 Tab 取代頁面堆疊式導航
4. 學習地圖（類多鄰國）取代題庫式列表
   - 學科天然可拆：年級 → 上/下學期 → 單元，結構清晰
   - 讓學習進度可視化，有路徑感與成就感
   - 現況問題：題庫風格 → 集題時間與維護成本高
   - 轉向：主題化選題，只挑有代表性、適合做遊戲的題目，不追求題海覆蓋
   - **決議：分兩階段**
     - 近期：章節式地圖（年級→學期→章節，每章是卡片，內含各模式題目+完成度）
     - 遠期：題量夠後再串成多鄰國式線性路徑
   - **一題多模式**：同一份考古題可同時支撐偵探/臥底/解密模式，三種玩法不重複但知識點強化三倍，有效放大題量
5. 讓遊戲更好玩 — 輔助線索收集冊
   - 現況：輔助線索只影響隱藏成就，找到也只顯示一句話，缺乏收集動機
   - 構想：線索收集冊（圖鑑系統）
     - 每條輔助線索 = 一張收集卡，不只是物件而是有意義的知識卡片
     - 卡片正面：收集到的線索物品/圖示
     - 卡片背面：對應的知識點，可用於複習
   - 雙重價值：收集成就感（遊戲動機）+ 知識複習工具（學習價值）
   - **決議：先做 localStorage 簡易版**
     - 跨題目永久圖鑑，資料結構 = `Set<cardId>`
     - UI 標註提示（如「目前資料儲存在本機，登入後可雲端同步」），讓試玩者理解現狀
     - 代碼中明確註解 `// TODO: 帳號系統上線後遷移至雲端`
     - 遷移成本極低，屆時直接搬
     - 圖鑑頁面與學習地圖整合，章節卡片顯示收集進度
6. 即時反饋感不足 — 缺乏「爽感」演出
   - 現況：互動反饋太平淡，沒有多鄰國那種即時震動/音效的刺激感
   - Web 震動限制：Vibration API 支援度有限，包成 App 前可能先略過
   - 構想：逆轉裁判風格的「異議！」演出
     - 全屏或螢幕中央橫向衝擊動畫 + 音效
     - 適用時機待定：找到關鍵線索？舉證成功？指認正確？階段轉場？
   - 目標：在關鍵節點製造「高光時刻」，讓學生有記憶點和成就爆發感
   - **決議：精準節點 + 不濫用**
     - 全屏演出：破案成功、Game Over（僅此兩個最重的）
     - 中型演出：最後一條線索收齊（橫幅）
     - 輕量確認：舉證成功（短促動畫）
     - 每次找到線索不演出，避免打斷節奏
   - **實作方案**
     - 文字衝擊演出：CSS 自幹（閃白→slide-in→shake），掌控主題適配
     - 特效點綴：Lottie 動畫（粒子/光暈），輕量且向量不失真
     - 音效：freesound.org / Pixabay（CC0），預載短音效 <50KB
     - 封裝為 `<CutsceneOverlay />` 單一元件，接受文字+主題參數
